    TLS Setup¶

    WitFoo Analytics uses TLS to encrypt all client-facing traffic. The reverse proxy terminates HTTPS connections and forwards requests to backend services over the internal Docker network.

    TLS Architecture¶

    graph LR
    CLIENT[Client Browser] -->|HTTPS :443| RP[Reverse Proxy]
    RP -->|HTTP :8080| API[API]
    RP -->|HTTP :3000| UI[UI]
    RP -->|HTTP :8003| DISP[Dispatcher]
    RP -->|HTTP :8082| IE[Incident Engine]
    CONDUCTOR[Conductor] -->|HTTPS :8443| AI[Artifact Ingestion]

    TLS is terminated at two points:

    1. Reverse Proxy (port 443) — All user-facing HTTPS traffic
    2. Artifact Ingestion (port 8443) — Conductor-to-Analytics artifact delivery

    Internal Traffic Is Unencrypted

    Traffic between services on the witfoo-analytics-net Docker network is not encrypted. This is acceptable when all services run on the same host. For multi-host deployments, use an encrypted overlay network or a service mesh.

    Certificate Requirements¶

    Attribute Requirement
    Format PEM-encoded X.509
    Key type RSA 2048+ or ECDSA P-256+
    Chain Full chain including intermediates
    SAN Must include the server's FQDN and/or IP
    Validity Recommended: 90 days (for automated renewal)

    Certificate File Locations¶

    Service Certificate Path Key Path
    Reverse Proxy ./certs/server.crt ./certs/server.key
    Artifact Ingestion /certs/server.crt /certs/server.key

    These paths are configured via environment variables:

    reverse-proxy:
  environment:
    - TLS_ENABLED=true
    - TLS_PORT=443
    - CERT_FILE=./certs/server.crt
    - KEY_FILE=./certs/server.key

artifact-ingestion:
  environment:
    - TLS_ENABLED=true
    - TLS_PORT=8443
    - CERT_FILE=/certs/server.crt
    - KEY_FILE=/certs/server.key
  volumes:
    - ../certs:/certs

    Generating Self-Signed Certificates¶

    For development and evaluation environments:

    mkdir -p certs

# Generate a self-signed certificate valid for 365 days
openssl req -x509 -nodes -days 365 \
  -newkey rsa:2048 \
  -keyout certs/server.key \
  -out certs/server.crt \
  -subj "/CN=witfoo-analytics" \
  -addext "subjectAltName=DNS:localhost,DNS:witfoo-analytics,IP:127.0.0.1"

# Set restrictive permissions
chmod 600 certs/server.key
chmod 644 certs/server.crt

    Self-Signed Certificates

    Self-signed certificates trigger browser warnings and should only be used for development. Use certificates from a trusted CA (or an internal PKI) for production deployments.

    Using Let's Encrypt Certificates¶

    For internet-facing deployments with a public domain:

    # Install certbot
apt-get install certbot

# Obtain certificate (standalone mode)
certbot certonly --standalone -d analytics.example.com

# Copy to the certs directory
cp /etc/letsencrypt/live/analytics.example.com/fullchain.pem certs/server.crt
cp /etc/letsencrypt/live/analytics.example.com/privkey.pem certs/server.key
chmod 600 certs/server.key

    Automated Renewal¶

    Create a renewal hook to copy updated certificates:

    #!/bin/bash
# /etc/letsencrypt/renewal-hooks/deploy/witfoo-analytics.sh

DOMAIN="analytics.example.com"
CERT_DIR="/path/to/docker/certs"

cp /etc/letsencrypt/live/$DOMAIN/fullchain.pem $CERT_DIR/server.crt
cp /etc/letsencrypt/live/$DOMAIN/privkey.pem $CERT_DIR/server.key
chmod 600 $CERT_DIR/server.key

    Using Internal CA Certificates¶

    For enterprise environments with an internal PKI:

    # Place your CA-signed certificate and key
cp /path/to/signed-cert.pem certs/server.crt
cp /path/to/private-key.pem certs/server.key

# If your CA provides an intermediate chain, concatenate it
cat /path/to/signed-cert.pem /path/to/intermediate-ca.pem > certs/server.crt

    Hot-Reload of Certificates¶

    The reverse proxy supports hot-reload of TLS certificates. When certificate files are updated on disk, the service detects the change and loads the new certificates without requiring a container restart.

    To trigger a certificate reload:

    1. Replace the certificate and key files in the certs/ directory
    2. The reverse proxy's file watcher detects the change
    3. New TLS connections use the updated certificate

    If hot-reload is not available or you need to force a reload:

    # Restart only the reverse proxy
docker compose -f docker-compose.aio.yml restart reverse-proxy

    Key File Permissions

    Ensure the private key file (server.key) has permissions 600 and is owned by the container's runtime user. The reverse proxy runs as a non-root user (UID 1000).

    Disabling TLS¶

    For environments where TLS is terminated by an external load balancer:

    TLS_ENABLED=false docker compose -f docker-compose.aio.yml up -d

    Or in your .env file:

    TLS_ENABLED=false

    When TLS is disabled, the reverse proxy serves HTTP only on port 8081.

    Never Disable TLS Without External Termination

    If you disable TLS on the reverse proxy, ensure that an upstream load balancer or WAF provides HTTPS termination. Serving the Analytics UI over plain HTTP exposes authentication tokens and sensitive data to network interception.

    TLS Environment Variables¶

    Variable Service Default Description
    TLS_ENABLED reverse-proxy, artifact-ingestion true Enable/disable TLS
    TLS_PORT reverse-proxy 443 HTTPS listen port
    CERT_FILE reverse-proxy, artifact-ingestion ./certs/server.crt Certificate file path
    KEY_FILE reverse-proxy, artifact-ingestion ./certs/server.key Private key file path

    Verifying TLS Configuration¶

    After deployment, verify the TLS setup:

    # Check certificate details
openssl s_client -connect localhost:443 -servername analytics.example.com < /dev/null 2>/dev/null | \
  openssl x509 -noout -subject -dates -issuer

# Verify the health endpoint over HTTPS
curl -k https://localhost/health

# Test with certificate verification (requires trusted CA)
curl --cacert /path/to/ca-cert.pem https://analytics.example.com/health

    Cassandra TLS¶

    Cassandra uses SSL for CQL client connections. The health check uses cqlsh --ssl:

    healthcheck:
  test: ["CMD-SHELL", "cqlsh --ssl -u cassandra -p cassandra -e 'DESCRIBE KEYSPACES' || exit 1"]

    Cassandra TLS certificates are managed separately from the reverse proxy certificates. For clustered deployments, configure inter-node encryption in the Cassandra configuration.
