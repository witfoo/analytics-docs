    WFA Integration¶

    In production environments, WitFoo Analytics is deployed and managed by the WitFoo Appliance (WFA) daemon. WFA handles container lifecycle, configuration generation, hardware validation, and upgrades. Docker Compose files serve as reference templates; WFA creates and manages containers programmatically.

    WFA Deployment Roles¶

    The wfa configure wizard presents six roles:

    # Role Node Role ID RAM CPU Description
    1 Conductor 12 8 GB 4 cores Central coordination and management
    2 Console 14 8 GB 4 cores Web UI console (no processing)
    3 AIO + Conductor 16 32 GB 8 cores Single-node with signal collection
    4 AIO 11 16 GB 4 cores Single-node Analytics only
    5 Processing 17 16 GB 4 cores Clustered compute without storage
    6 Data 5 16 GB 4 cores Clustered Cassandra storage

    Installation Workflow¶

    Step 1: Configure the Node¶

    Run the interactive configuration wizard:

    wfa configure

    The wizard prompts for:

    • Deployment role — Select from the six roles above
    • License configuration — Enter your WitFoo license key
    • Network settings — IP addresses, hostname, DNS
    • Certificate settings — TLS certificate paths or auto-generation
    • Web UI port and protocol — HTTPS port (default: 443)
    • Admin credentials — Initial administrator username and password
    • Optional features — Metrics collection, offline mode, auto-update
    • Clustering settings — Seed nodes and data node addresses (for clustered roles)

    The wizard generates two configuration files:

    File Purpose
    /witfoo/configs/node.json Node identity, networking, certificates
    /witfoo/configs/analytics.json Analytics-specific settings (role, retention, modules)

    Post-Configuration Settings

    Data retention, UI modules, and CyberGrid settings are no longer configured during wfa configure. These are initialized with sensible defaults and customized later via the Onboarding Wizard in the Analytics web UI.

    Step 2: Validate Hardware¶

    wfa analytics install

    This command validates that the host meets minimum hardware requirements for the configured role. If the system does not meet requirements, the command exits with an error and a description of what is insufficient.

    Step 3: Start Services¶

    wfa start

    The WFA daemon:

    1. Pulls container images for the configured role
    2. Creates containers with the correct environment variables
    3. Starts services in dependency order (Cassandra → NATS → processing → application)
    4. Waits for health checks to pass

    Step 4: Verify Status¶

    wfa analytics status

    Lists all expected services for the configured role and their current state:

    Service                    Status      Container
─────────────────────────────────────────────────
cassandra                  running     witfoo-analytics-cassandra
nats                       running     witfoo-analytics-nats
artifact-ingestion         running     witfoo-analytics-artifact-ingestion
graph-processor            running     witfoo-analytics-graph-processor
incident-engine            running     witfoo-analytics-incident-engine
api                        running     witfoo-analytics-api
dispatcher                 running     witfoo-analytics-dispatcher
reverse-proxy              running     witfoo-analytics-reverse-proxy
ui                         running     witfoo-analytics-ui
prometheus                 running     witfoo-analytics-prometheus
grafana                    running     witfoo-analytics-grafana

    Configuration Files¶

    node.json¶

    Generated by wfa configure, this file defines the node's identity and network configuration:

    {
  "node_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "hostname": "analytics-01.example.com",
  "role": 11,
  "ip_address": "10.0.1.10",
  "certificates": {
    "cert_file": "/witfoo/certs/server.crt",
    "key_file": "/witfoo/certs/server.key"
  }
}

    analytics.json¶

    Analytics-specific configuration:

    {
  "role": "analytics-aio",
  "retention": {
    "artifacts_days": 7,
    "work_units_days": 365,
    "work_collections_days": 365,
    "reports_days": 1825
  },
  "ui_modules": "all",
  "cybergrid": {
    "enabled": false,
    "publish_enabled": false
  },
  "clustering": {}
}

    Clustering Configuration (Data Node)¶

    {
  "role": "analytics-data-node",
  "clustering": {
    "seed_nodes": ["10.0.1.10", "10.0.1.11"],
    "listen_address": "10.0.1.10"
  }
}

    Clustering Configuration (Processing Node)¶

    {
  "role": "analytics-processing-node",
  "clustering": {
    "data_node_addresses": ["10.0.1.10:9042", "10.0.1.11:9042"]
  }
}

    Upgrade Procedure¶

    1. Validate the Upgrade¶

    wfa analytics upgrade --branch v7.1.0

    This checks compatibility and downloads upgrade metadata without applying changes.

    2. Apply the Upgrade¶

    wfa stop
wfa start

    The WFA daemon automatically pulls new images on the next start.

    3. Verify¶

    wfa analytics status

    Backup Before Upgrading

    Always snapshot Cassandra data volumes before performing upgrades. Use nodetool snapshot or volume-level snapshots depending on your infrastructure.

    CLI Reference¶

    Command Description
    wfa configure Interactive configuration wizard
    wfa analytics install Validate hardware and prepare for deployment
    wfa start Start all services for the configured role
    wfa stop Stop all running services
    wfa analytics status Show service status for the current role
    wfa analytics upgrade --branch <version> Validate and prepare an upgrade

    WFA vs. Docker Compose¶

    Capability WFA Docker Compose
    Production deployment ✅ ❌ (dev/CI only)
    Hardware validation ✅ ❌
    Automatic dependency ordering ✅ ✅
    Configuration wizard ✅ ❌ (manual .env)
    Rolling upgrades ✅ ❌
    Certificate management ✅ Manual
    Appliance images ✅ ❌

    For appliance downloads and installation scripts, see the witfoo-appliances repository.
