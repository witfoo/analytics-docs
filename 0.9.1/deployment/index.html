    Deployment Overview¶

    WitFoo Analytics v0.9.1 is an enterprise security operations platform deployed as a set of containerized microservices. This guide covers production deployment for DevOps engineers using either the WitFoo Appliance (WFA) daemon or Docker Compose.

    Architecture Overview¶

    WitFoo Analytics consists of interconnected services organized into three tiers: foundation (data and messaging), processing (analytics pipeline), and application (API, UI, and routing).

    Deployment Roles¶

    WitFoo Analytics supports four deployment roles to match different infrastructure requirements.

    Role Services Description
    AIO + Conductor 15 All Analytics and Conductor services on a single node
    AIO 11 All Analytics services without Conductor
    Data Node 1 Dedicated Cassandra node for clustered storage
    Processing Node 10 All services except Cassandra; connects to external data nodes

    AIO (Analytics Only)¶

    Resource Minimum Recommended

    RAM 16 GB 32 GB
    Disk 200 GB SSD 1 TB SSD

    Data Node¶

    Resource Minimum Recommended
    CPU 4 cores 8 cores
    RAM 16 GB 32 GB
    Disk 500 GB SSD 2 TB SSD

    Processing Node¶

    Resource Minimum Recommended
    CPU 4 cores 16 cores
    RAM 16 GB 32 GB
    Disk 100 GB SSD 500 GB SSD

    Disk Requirements

    All roles require SSD storage. Cassandra performance degrades significantly on spinning disks. Data nodes in production should use dedicated NVMe drives.

    Network Ports¶

    Port Protocol Service Direction
    443 TCP Reverse Proxy (HTTPS) Inbound
    8081 TCP Reverse Proxy (HTTP) Inbound
    8080 TCP API Internal
    8082 TCP Incident Engine Internal
    8090 TCP Artifact Ingestion Internal
    3000 TCP UI Internal
    9042 TCP Cassandra CQL Internal
    7000 TCP Cassandra inter-node Internal (clustered)
    4222 TCP NATS client Internal
    6222 TCP NATS cluster Internal
    8222 TCP NATS monitoring Internal
    9090 TCP Prometheus Localhost
    3001 TCP Grafana Localhost

    Production Firewall Rules

    Only port 443 (HTTPS) should be exposed to end users. Bind all internal services to 127.0.0.1 or a private network interface using the LISTEN_IP environment variable.

    Prerequisites¶

    • Docker Engine 24.0+ with Compose V2
    • Linux host (Ubuntu 22.04+ or RHEL 9+ recommended)
    • Git credentials for private image builds (stored at $HOME/.git-credentials)
    • TLS certificates for production HTTPS (see TLS Setup)

    Quick Start¶

    For a single-node evaluation deployment:

    cd docker

# Start AIO stack
docker compose -f docker-compose.aio.yml up -d --build

# Wait for all services to become healthy
./scripts/ci/wait-for-services.sh \
  --compose-file docker/docker-compose.aio.yml \
  --timeout 300

# Verify
docker compose -f docker-compose.aio.yml ps

    Access the platform at https://localhost (self-signed certificate) or http://localhost:8081.
