    Configuration Wizard¶

    The sudo wfa configure command launches an interactive wizard that configures your WitFoo Appliance. This page documents every step of the wizard, including the exact prompts, options, and role-specific differences.

    Overview¶

    The wizard collects configuration for your appliance and writes the results to /witfoo/configs/node.json. The steps vary depending on the selected role.

    Step Applicability by Role¶

    Step Conductor Console Analytics
    1. Organization Information ✅ ✅ ✅
    2. Role Selection ✅ ✅ ✅
    3. License Configuration ✅ ✅ ✅
    4. Appliance ID ✅ — ✅
    5. Node Hostname and IP ✅ ✅ ✅
    6. Network Configuration ✅ — —
    7. Registry Configuration ✅ ✅ ✅
    8. Certificate Configuration ✅ ✅ ✅
    9. Admin Password ✅ ✅ ✅
    10. Optional Features ✅ ✅ ✅
    11. Review and Save ✅ ✅ ✅

    Running the Wizard¶

    Connect to your appliance via SSH and run:

    sudo wfa configure

    The wizard presents each step sequentially. You can accept defaults (shown in brackets) by pressing Enter, or type a new value.

    Step 1: Organization Information¶

    The wizard begins by collecting your organization details.

    === Organization Information ===

Organization ID [witfoo]: mycompany
Organization Name [WitFoo]: My Company Inc.

    Organization ID — A DNS-valid, lowercase identifier for your organization. This is used internally for data partitioning and multi-tenant isolation. Use only lowercase letters, numbers, and hyphens.

    Organization Name — The human-readable display name for your organization. This appears in the web UI and reports.

    Organization ID Format

    The Organization ID must be DNS-valid: lowercase letters, numbers, and hyphens only. No spaces, underscores, or special characters. Examples: acme-corp, mycompany, security-ops.

    Step 2: Role Selection¶

    Select the role for this appliance.

    === Role Selection ===

Select the role for this appliance:

  1. Conductor   — Data ingestion and signal processing (4 CPU / 8 GB RAM min)
  2. Console     — Remote management and monitoring    (4 CPU / 8 GB RAM min)
  3. Analytics   — Security analytics platform         (8 CPU / 12 GB RAM min)

Role [3]:

    After selecting a role, the wizard checks your hardware against the minimum requirements:

    Role CPU (min) RAM (min) Disk (min)
    Conductor 4 8 GB 220 GB
    Console 4 8 GB 220 GB
    Analytics 8 12 GB 220 GB

    If your hardware is below the minimum, the wizard displays a warning:

    ⚠ WARNING: This system has 4 CPU cores. The Analytics role requires a minimum
  of 8 CPU cores. Performance may be degraded. Continue anyway? [y/N]:

    Recommended Hardware for Analytics

    While the minimum for Analytics is 8 CPU / 12 GB RAM, production deployments should use 16 CPU cores, 32 GB RAM, and 1 TB disk for optimal performance with Cassandra, the graph processor, and the incident engine.

    Step 3: License Configuration¶

    The wizard configures your WitFoo license.

    Cloud Deployments¶

    For cloud marketplace deployments, the license is fetched automatically:

    === License Configuration ===

Fetching license from licensing.witfoo.com...
License validated: My Company Inc. (Enterprise, 500 cores)

    On-Premises Deployments¶

    For on-premises deployments, you have two options:

    === License Configuration ===

  1. Enter license key
  2. Request 15-day trial

Selection [1]:

    Option 1: Enter License Key

    License Key: XXXX-XXXX-XXXX-XXXX
Validating license... ✓ License valid: My Company Inc. (Enterprise, 500 cores)

    Option 2: Request 15-Day Trial

    Company Name: My Company Inc.
Contact Name: Jane Smith
Contact Email: jane.smith@mycompany.com
Core Count [16]: 16

Requesting trial license... ✓ Trial license activated (expires: 2026-03-15)

    Trial Limitations

    The 15-day trial provides full platform functionality. Contact WitFoo sales to obtain a production license before the trial expires.

    Step 4: Appliance ID¶

    An appliance ID is automatically generated for Conductor and Analytics nodes. This UUID uniquely identifies the appliance in multi-node deployments.

    === Appliance ID ===

Appliance ID: a1b2c3d4-e5f6-7890-abcd-ef1234567890

    Console Skips This Step

    The Console role does not require an Appliance ID and skips this step automatically.

    Step 5: Node Hostname and IP¶

    Configure the network identity of this node.

    === Node Hostname and IP ===

Hostname [wfa-analytics-01]: wfa-analytics-01
Detected IP address: 10.0.1.50
Use detected IP? [Y/n]: y
IP Address: 10.0.1.50

    If the auto-detected IP is incorrect (e.g., the system has multiple interfaces), select n and enter the correct IP manually:

    Use detected IP? [Y/n]: n
IP Address: 192.168.1.100

    Step 6: Network Configuration (Conductor Only)¶

    This step configures the NATS broker ports and data limits for the Conductor's signal processing pipeline.

    === Network Configuration ===

NATS Client Port [4223]: 4223
NATS Leaf Port [4443]: 4443
Max Data Size (MB) [1024]: 1024
    Setting Default Description
    NATS Client Port 4223 Port for client connections to the NATS broker
    NATS Leaf Port 4443 Port for leaf node connections from other NATS nodes
    Max Data Size 1024 MB Maximum data payload size for NATS messages

    Console and Analytics Skip This Step

    Network configuration for the NATS broker is only relevant to the Conductor role. Console and Analytics nodes skip this step.

    Step 7: Registry Configuration¶

    Configure credentials for the container image registry. These credentials are typically populated automatically from your license validation.

    === Registry Configuration ===

Registry URL [registry.witfoo.com]: registry.witfoo.com
Registry Username: mycompany
Registry Password: ********
Validating registry credentials... ✓ Authenticated

    The registry credentials allow the WFA daemon to pull container images for the services required by your selected role.

    Step 8: Certificate Configuration¶

    Optionally configure TLS certificates for secure communication. If you skip this step, the appliance generates self-signed certificates.

    === Certificate Configuration ===

Configure custom TLS certificates? [y/N]: y

Client Certificate Path [/witfoo/certs/client.crt]: /witfoo/certs/client.crt
Client Key Path [/witfoo/certs/client.key]: /witfoo/certs/client.key
Server Certificate Path [/witfoo/certs/server.crt]: /witfoo/certs/server.crt
Server Key Path [/witfoo/certs/server.key]: /witfoo/certs/server.key
CA Certificate Path [/witfoo/certs/ca.crt]: /witfoo/certs/ca.crt

Validating certificates... ✓ Certificates valid
    Certificate Default Path Purpose
    Client Certificate /witfoo/certs/client.crt Client-side TLS authentication
    Client Key /witfoo/certs/client.key Client certificate private key
    Server Certificate /witfoo/certs/server.crt Server-side TLS for HTTPS
    Server Key /witfoo/certs/server.key Server certificate private key
    CA Certificate /witfoo/certs/ca.crt Certificate authority chain

    Production Deployments

    Self-signed certificates are acceptable for evaluation but should be replaced with certificates from a trusted CA for production deployments. Browsers will display security warnings for self-signed certificates.

    Step 9: Admin Password¶

    Set the administrative password for the appliance. This password is used for local appliance management.

    === Admin Password ===

Enter admin password (min 8 characters): ********
Confirm admin password: ********
✓ Password saved
    • Minimum 8 characters
    • Input is masked (not displayed)
    • Confirmation is required
    • The password is saved to /data/local/.passwd

    Choose a Strong Password

    This password provides administrative access to the appliance. Use a strong, unique password that is at least 12 characters with a mix of uppercase, lowercase, numbers, and symbols.

    Step 10: Optional Features¶

    The optional features presented depend on the selected role.

    Conductor Features¶

    === Optional Features ===

Enable local metrics (Prometheus/Grafana)? [Y/n]: y
Enable offline mode? [y/N]: n
Enable metrics export? [y/N]: n
Enable auto-update? [Y/n]: y
Remote Console FQDN []: console.mycompany.com
    Feature Default Description
    Local metrics Enabled Run Prometheus and Grafana locally for pipeline monitoring
    Offline mode Disabled Operate without internet connectivity (requires pre-staged images)
    Metrics export Disabled Export metrics to an external Prometheus/Grafana instance
    Auto-update Enabled Automatically pull and apply container image updates
    Remote Console FQDN Empty FQDN of the Console node managing this Conductor

    Console Features¶

    === Optional Features ===

Enable offline mode? [y/N]: n
Enable auto-update? [Y/n]: y
    Feature Default Description
    Offline mode Disabled Operate without internet connectivity
    Auto-update Enabled Automatically pull and apply updates

    Analytics Features¶

    === Optional Features ===

=== Data Retention ===
Artifact retention (days) [7]: 7
Work unit retention (days) [365]: 365
Work collection retention (days) [365]: 365
Report retention (days) [1825]: 1825

=== UI Module Visibility ===
Enable all modules (Search, Observer, Reporter, CyberGrid, AI)? [Y/n]: y

=== CyberGrid Integration ===
Enable CyberGrid threat intelligence? [y/N]: n

=== Clustering ===
Enable clustering? [y/N]: n
    Feature Default Description
    Data retention 7/365/365/1825 days Per-feature retention periods for artifacts, work units, collections, and reports
    UI module visibility All enabled Toggle which modules appear in the web UI navigation
    CyberGrid integration Disabled Participate in WitFoo's threat intelligence sharing network
    Clustering Disabled Enable multi-node Cassandra clustering for horizontal scaling

    Analytics Defaults Are Sensible

    The default retention and module settings work well for most deployments. These settings can also be customized later via the Onboarding Wizard in the web UI (see First Login & Onboarding).

    Step 11: Review and Save¶

    The wizard displays a summary of all configured settings for your review.

    === Configuration Summary ===

Organization:     My Company Inc. (mycompany)
Role:             Analytics
License:          Enterprise (500 cores, expires 2027-01-15)
Appliance ID:     a1b2c3d4-e5f6-7890-abcd-ef1234567890
Hostname:         wfa-analytics-01
IP Address:       10.0.1.50
Registry:         registry.witfoo.com
TLS Certificates: Custom (/witfoo/certs/)
Auto-Update:      Enabled

Data Retention:
  Artifacts:         7 days
  Work Units:        365 days
  Work Collections:  365 days
  Reports:           1825 days

UI Modules:       All enabled
CyberGrid:        Disabled
Clustering:       Disabled

Save configuration? [Y/n]: y

✓ Configuration saved to /witfoo/configs/node.json
✓ Starting services...

    After confirming, the wizard:

    1. Writes the configuration to /witfoo/configs/node.json
    2. Pulls the required container images from the registry
    3. Starts all services for the configured role

    Re-Running the Wizard

    You can re-run sudo wfa configure at any time to change your configuration. The wizard pre-populates fields with your existing settings.

    Configuration File Reference¶

    The wizard generates /witfoo/configs/node.json with the following structure:

    {
  "org_id": "mycompany",
  "org_name": "My Company Inc.",
  "role": "analytics",
  "appliance_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "hostname": "wfa-analytics-01",
  "ip_address": "10.0.1.50",
  "registry": {
    "url": "registry.witfoo.com",
    "username": "mycompany"
  },
  "certificates": {
    "client_cert": "/witfoo/certs/client.crt",
    "client_key": "/witfoo/certs/client.key",
    "server_cert": "/witfoo/certs/server.crt",
    "server_key": "/witfoo/certs/server.key",
    "ca_cert": "/witfoo/certs/ca.crt"
  },
  "features": {
    "auto_update": true,
    "offline_mode": false
  }
}
