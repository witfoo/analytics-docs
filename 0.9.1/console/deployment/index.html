    Deployment Guide¶

    This guide covers deploying WitFoo Console v0.9.1 in production environments using Docker Compose, including configuration, TLS termination, health monitoring, and upgrade procedures.

    Prerequisites¶

    Requirement Minimum Version
    Docker 20.10+
    Docker Compose 2.0+
    Available Port 8000 (configurable)
    Disk Space 500 MB (application + data)
    Memory 256 MB RAM

    Docker Compose Deployment¶

    1. Create the Deployment Directory¶

    mkdir -p /opt/witfoo-console
cd /opt/witfoo-console

    2. Create docker-compose.yml¶

    version: '3.8'

services:
  console-ui:
    image: ghcr.io/witfoo-dev/console-ui:main
    container_name: console-ui
    ports:
      - "8000:8000"
    volumes:
      - ./data:/data
    environment:
      - SECRET_KEY=<your-64-byte-hex-encoded-secret>
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "wget", "--quiet", "--tries=1", "--spider", "http://localhost:8000/api/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 10s

    3. Generate a Secret Key¶

    export SECRET_KEY=$(openssl rand -hex 32)
echo "SECRET_KEY=${SECRET_KEY}" >> .env

    4. Set a Custom Admin Password (Optional)¶

    mkdir -p data
echo "YourSecureAdminPassword" > data/.passwd

    5. Start the Service¶

    docker-compose up -d

    6. Verify Deployment¶

    # Check container status
docker-compose ps

# Check health
curl -s http://localhost:8000/api/health

# View logs
docker-compose logs -f

    Console is now accessible at http://localhost:8000.

    Important

    Change the default admin password immediately after first login. The default credentials are:

    • Username: admin
    • Password: Value from data/.passwd or F00theN0ise!

    Environment Variables¶

    Variable Required Default Description
    SECRET_KEY Yes (production) Development default 64-byte hex-encoded session encryption key

    Create a .env file alongside docker-compose.yml for environment variable management:

    SECRET_KEY=a1b2c3d4e5f6...  # output of openssl rand -hex 32

    Database Initialization and Migration¶

    Console handles database lifecycle automatically:

    flowchart TD
    A[Server Starts] --> B{data/ directory exists?}
    B -->|No| C[Create data/ directory]
    B -->|Yes| D{console.db exists?}
    C --> D
    D -->|No| E[Create console.db]
    D -->|Yes| F[Run migrations]
    E --> F
    F --> G{Users exist?}
    G -->|No| H[Create default admin user<br/>Create mssp-admin org]
    G -->|Yes| I[Server ready]
    H --> I
    • Database file: data/console.db
    • Migrations: Applied automatically on every startup
    • Default organization: mssp-admin created on first launch
    • Default user: admin with password from data/.passwd or F00theN0ise!

    Data Persistence

    The data/ directory must be mounted as a Docker volume to persist data across container restarts. Without a volume mount, all data is lost when the container is removed.

    Reverse Proxy and TLS Termination¶

    For production deployments, place Console behind a TLS-terminating reverse proxy.

    Nginx Example¶

    server {
    listen 443 ssl http2;
    server_name console.example.com;

    ssl_certificate     /etc/ssl/certs/console.example.com.pem;
    ssl_certificate_key /etc/ssl/private/console.example.com.key;
    ssl_protocols       TLSv1.2 TLSv1.3;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

server {
    listen 80;
    server_name console.example.com;
    return 301 https://$host$request_uri;
}

    Important

    WebSocket support is required for real-time node status updates. Ensure your reverse proxy configuration includes the Upgrade and Connection headers as shown above.

    Caddy Example¶

    console.example.com {
    reverse_proxy localhost:8000
}

    Caddy automatically provisions TLS certificates via Let's Encrypt.

    Health Check Endpoint¶

    Console exposes a health check endpoint for container orchestration and monitoring:

    GET /api/health

    Healthy Response: 200 OK

    The Docker Compose health check is configured to poll this endpoint every 30 seconds with a 10-second timeout and 3 retries after a 10-second startup grace period.

    Monitoring Integration¶

    Use the health endpoint with external monitoring systems:

    # Nagios/Icinga check
curl -sf http://console:8000/api/health || exit 2

# Prometheus blackbox exporter target
- targets:
  - http://console:8000/api/health

    Upgrading¶

    Standard Upgrade Procedure¶

    1. Back up the database:

      docker-compose stop
cp -r data/ data-backup-$(date +%Y%m%d)/

    2. Pull the new image:

      docker-compose pull

    3. Start the updated service:

      docker-compose up -d

    4. Verify the upgrade:

      docker-compose logs -f
curl -s http://localhost:8000/api/health

    5. Test functionality:

      • Log in to the Console UI
      • Verify organizations, nodes, and users are intact
      • Confirm node connectivity

    Rollback Procedure¶

    If an upgrade fails:

    # Stop the current version
docker-compose down

# Restore the database backup
rm -rf data/
cp -r data-backup-YYYYMMDD/ data/

# Pin the previous image version in docker-compose.yml
# image: ghcr.io/witfoo-dev/console-ui:previous-branch

# Start the previous version
docker-compose up -d

    Important

    Database migrations may not be reversible. Always back up the data/ directory before upgrading.

    Maintenance Procedures¶

    Database Backup¶

    # Stop the service for a consistent backup
docker-compose stop
cp data/console.db data/console.db.backup
docker-compose start

# Or use sqlite3 for online backup (if available)
sqlite3 data/console.db ".backup data/console.db.backup"

    Log Management¶

    Console logs to stdout/stderr. Configure Docker's logging driver for log rotation:

    services:
  console-ui:
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

    Container Resource Limits¶

    For production deployments, set resource limits:

    services:
  console-ui:
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '1.0'
        reservations:
          memory: 128M
          cpus: '0.25'
